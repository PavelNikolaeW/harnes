Архитектура универсального автономного агента

Версия 0.2 · зафиксировано 2026-05-26 · статус: stack-freeze v0

Документ фиксирует архитектуру обвязки над локальными LLM для исследования гипотезы о метацикле над ReAct-циклом. Цель — система, которая по целям из не заданного на этапе проектирования распределения доводит их до завершения в консеквенциальном пространстве действий на протяжённом горизонте при минимальном пошаговом контроле оператора.

Документ описывает компоненты, схему взаимодействия и зафиксированные решения. Открытые вопросы перечислены в финальном разделе.

1. Постановка задачи #

Объект. Универсальный автономный агент — система, которая по целям из открытого, не заданного на этапе проектирования распределения доводит их до завершения в консеквенциальном пространстве действий на протяжённом горизонте при минимальном пошаговом контроле оператора. Операциональные определения:

  • «универсальный» — относительно распределения задач (оценка на отложенных и заведомо новых средах);
  • «автономный» — положение на спектре «локус и зернистость контроля» (доля вмешательств на единицу работы, длина горизонта без коррекции);
  • «агент» — действует на среду, меняя её состояние, в отличие от системы, лишь выдающей предсказание или текст.

Пробел. Каждое из трёх свойств по отдельности достигнуто. Открыт вопрос их конъюнкции: что позволяет одной системе сохранять компетентное и робастное преследование цели по мере расширения распределения задач и удлинения горизонта. Это разворачивается в две измеримые оси — ширина распределения и длина горизонта.

Гипотеза. Метацикл поверх ReAct-цикла, работающий непрерывно (не ждёт сообщений оператора), с явными этапами восприятия, внимания, арбитража целей, recall'а, исполнения, верификации, обновления явной модели мира, рефлексии и сохранения. Надёжность — метрика первого класса наряду со способностью.

Критерий прогресса. Производительность на отложенном распределении задач при ограниченном бюджете человеческих вмешательств на длинном горизонте. Без held-out-оценки заявление об общности нефальсифицируемо.

Это исследовательская система. Стоимость токенов и латентность инференса не оптимизируются. Решения принимаются в пользу методологической чистоты и проверяемости гипотезы.

2. Обзор архитектуры #

Два вложенных контура:

  • Метацикл — внешний синхронный цикл, тикает непрерывно. Управляет вниманием, целями, памятью, рефлексией.
  • ReAct-цикл — внутренний, запускается метациклом под активную листовую цель. Делает тактическое планирование и исполнение в пределах одной траектории.

Двухуровневая декомпозиция работы

  • Goal decomposition (в goal_arbitration): разложение цели на подцели. Подцели — это Goal-объекты со своими предикатами и бюджетами, живут на длинном горизонте.
  • Tactical planning (внутри ReAct): разложение листовой цели на последовательность действий внутри одной траектории. Шаги плана не цели, они транзиентны.

Общий поток одного тика

[tick]
  sense                    — polling + чтение event-queue, без LLM
  attend                   — score(relevance, novelty, urgency);
                             прогон standing-целей по FocusFrame
  goal_arbitration         — арбитраж + источник новых целей
                             + проверка pending_verifications
                             + operator-approval flow для top-level self-целей
                             → active_goal | idle
  if active_goal:
    recall(coarse)         — единый memory layer (vector + graph + log)
    react_loop             — внутренний цикл, см. § 7
    verify(immediate)      — leaf-level; composite/external → pending_verifications
    world_update           — обновление явной модели мира (KG, темпоральный)
    if reflect_trigger:
      reflect              — режим выводится из триггера
    store                  — маршрутизация в бэкенды памяти
[end tick]

3. Метацикл #

Стадии метацикла как чистые функции от своего входа в выход. Состояние явное, проходит через тик-стейт. Этапы вызываются последовательно прямыми функциональными вызовами (не blackboard, не event-bus).

Этап Назначение и поведение Частота
sense Сборка сырого состояния среды и внутренних триггеров. Polling сенсоров + чтение event-queue (push). Lossless в raw-лог. LLM-вызовов нет. Выход: ObservationBundle. Каждый тик
attend Из ObservationBundle отобрать релевантное для активной цели, проранжировать. Score = взвешенная сумма relevance (cosine эмбеддинга наблюдения и описания цели) + novelty (расстояние от recent attention history) + urgency (теги источника + паттерны оператора). Здесь же — прогон standing-целей по результату. Выход: FocusFrame. Каждый тик
goal_arbitration Стейтфул-сервис управления целями. Арбитраж активной цели по приоритету и дедлайнам. Источник новых целей (декомпозиция родителя, аномалия в attend, maintenance-расписание). Проверка очереди pending_verifications. Operator-approval flow для top-level self-целей. Выход: active_goal | idle + allowed_skills + budget. Каждый тик
recall (coarse) Грубое вспоминание для прайминга ReAct'а. Запросы по трём индексам (vector / graph / episodic-log) с низким top-k. Тонкое доставание остаётся как retrieval-tool внутри ReAct. Выход: MemoryBundle. Перед запуском ReAct
react_loop Внутренний цикл, см. § 7. Вход: ActiveGoal, FocusFrame, MemoryBundle, allowed_skills. Выход: Trajectory. Когда есть active_goal
verify (immediate) Проверка predicate_of_success листовой цели по выходу ReAct. Анти-коррелирован с исполнителем: отдельный промпт, в идеале — отдельная модель. Composite/external верификации регистрируются в pending_verifications и резолвятся в последующих тиках. Выход: Verdict. После выхода ReAct
world_update Обновление явной модели мира на основе наблюдённых дельт состояния. Сравнение предсказаний (из Action.expected_outcome) с реальностью. Создание/обновление сущностей и связей в темпоральном KG. После verify
reflect Извлечение паттернов из окна опыта. Режим выбирается по триггеру (см. § 15). Может обновлять реестр скиллов, политики attention, стратегии, standing-цели. Дорогой LLM-вызов с большим контекстом. По триггеру
store Маршрутизация по бэкендам памяти: episodic-лог, vector-store, KG, реестр скиллов. Индексация, дедупликация, decay. Post-ReAct и post-reflect

Решения по метациклу

  1. Цикл синхронный, без прерываний (v1). Тик атомарный; новые события ждут в очереди до следующего тика. Длительность тика лечится бюджетом ReAct.
  2. Обмен данными между этапами — direct sequential calls. Тик-стейт прокидывается через этапы, параллельно эмитится trace-log для наблюдаемости.
  3. Push-события идут в очередь; sense читает её + делает active polling. Рестарта тика на push нет.
  4. Один ReAct в момент времени. Параллелизм — только через spawn_subagent внутри ReAct, и тот в v1 синхронный.
  5. Тяжёлая работа условна. Большинство тиков лёгкие: sense → attend → goal_arbitration → idle.

4. Подсистема целей #

Схема цели

Goal {
  id
  parent_id              # дерево декомпозиции
  depends_on: [id,...]   # DAG-зависимости между нодами (поверх дерева)

  description            # естественный язык
  class                  # task | inquiry | maintenance | standing | practice
  predicate_of_success   # типизированный, см. ниже

  priority
  deadline               # опционально

  status                 # pending | pending_approval | active | suspended
                         # | done | failed | abandoned
  budget {
    tokens, steps, time, money
    consumed, remaining
  }

  allowed_skills: [skill_id,...]

  aggregation            # для внутренних нод дерева: all | custom (v1)

  origin                 # operator | decomposition | self
  originator             # кто конкретно создал
  origin_subtype         # для self: curiosity | learning | maintenance | anticipatory

  created_at, updated_at
  metadata
}

Иерархия

Иерархия — дерево (один parent_id), но между нодами разрешены depends_on-связи (DAG поверх дерева). Это даёт почти всю выразительность полноценного DAG'а без его минусов (циклы, топосорт).

Бюджеты

Бюджеты независимы по целям. При создании ребёнка ему явно выдаётся бюджет; бюджет родителя на это не уменьшается (общий системный бюджет — отдельная сущность). Дефолты — оператор; LLM может предложить изменение, reflect — тоже, финальное слово оператор. Бюджет в схеме мутабельный с историей правок.

Predicate-types

ТипСемантикаКогда применять
structural Точное соответствие схеме/значению Самый сильный. Берётся первым выбором ради research-валидности.
state_change Свойство среды стало истинно после действия Сильный. Требует state-проверки во внешней среде.
judge LLM или человек судит по описанию Слабый, шумный. Метрика наследует ошибки судьи. Использовать с явным флагом «измерено LLM-судьёй».
composite Аггрегация над детьми Только для внутренних нод дерева. Аггрегация — all или custom в v1.
external Другой агент, система или время подтверждают Всегда deferred — регистрируется в pending_verifications.

Классы целей

КлассХарактерPredicateLifecycleПример
task Разовое достижение structural / state_change / judge One-shot, закрывается по completion «написать функцию X»
inquiry Открытый сбор знания judge или ручной критерий насыщения One-shot, может порождать sub-task'и «разобраться, как работает Y»
maintenance Системная гигиена structural (например, «размер episodic-лога < N») Recurring, перепланирует сам себя «консолидировать память»
standing Политика «всегда» Per-tick check предиката Always-active, нет состояния done «если verify=fail и priority≥P — поднять inquiry»
practice Тренировка скилла structural + метрика One-shot, выход = обновление метрики скилла «выполнить N раз skill X на синтетике»

Aggregation

В v1 поддерживаются all и custom. any, k-of-n, weighted — добавляются по факту появления случаев.

Origin и право на self-цели

Агент может ставить себе цели любого класса, но верхнеуровневые self-цели (parent_id is null AND origin = self) проходят через operator-approval (см. § 6). Декомпозиция уже активной цели — без апрува. Это явно фиксирует позицию на спектре «локус и зернистость контроля» в твоём определении автономности.

Верификация для дерева целей

  • Leaf verify — синхронный, сразу после exit ReAct'а.
  • Composite verify — для внутренней ноды; триггерится, когда все потомки в терминальном статусе. Либо аггрегация (all / custom), либо отдельная проверка родительского предиката.
  • Deferred verify — когда верификация зависит от внешнего события, другого агента или времени. Регистрируется в pending_verifications; на каждом тике goal_arbitration проверяет, появились ли в sense факты, позволяющие разрешить верификацию.

Решения по подсистеме целей

  1. Иерархия — tree + depends_on (DAG-lite).
  2. Бюджеты независимы по целям (вариант A).
  3. Predicate-types — все пять (structural, state_change, judge, composite, external); приоритет на structural и state_change для research-валидности.
  4. Aggregation в v1 — all и custom.
  5. Классы — все пять (task, inquiry, maintenance, standing, practice).
  6. Self-цели верхнего уровня — через operator-approval с эскалацией по таймауту.
  7. Verify трёх уровней: immediate-leaf / composite / deferred.

5. Standing-цели #

Standing-цели — постоянно активные условные политики (реактивный слой). Наблюдают состояние, порождают task-подцели при срабатывании условия. Сами никогда не «выполняются», не имеют статуса done, действий не совершают — только триггерят новые цели.

Примеры

  • «Когда в sense приходит alert от X — поднять task: investigate alert».
  • «Когда world_update зафиксировал, что сервис Y в нерабочем состоянии — поднять task: восстановить Y».
  • «Если последняя operator-цель завершена больше T тиков назад — поднять inquiry: что делать дальше».
  • «Если размер episodic-лога > N без консолидации — поднять maintenance: consolidate memory».
  • «Когда verify=fail на цели с приоритетом ≥ P — поднять inquiry: diagnose failure».

Что НЕ делается через standing-цели

  • Валидация аргументов тулов — это слой tool-layer (schema-validation).
  • Детекция ошибок исполнения — это verify; standing-цели могут лишь превращать сигнал verify=fail в новую подцель.

Проверяются в начале этапа attend: для каждой standing-цели — чек условия против FocusFrame; если выстрелило — породить task-подцель и поставить в очередь goal_arbitration.

Решения по standing-целям

  1. Количество standing-целей ограничено: 5–15 штук — «характер» агента.
  2. Если их добавляет reflect — есть потолок M; новая standing-цель должна вытолкнуть менее эффективную по метрике «срабатываний и пользы».
  3. Standing-цели только триггерят, не действуют сами.

6. Operator-approval flow #

Применяется только к top-level self-целям (parent_id is null AND origin = self).

агент порождает цель
  → создаётся со status = pending_approval
  → оператор получает уведомление (лог / Slack / другой канал)

оператор может:
  - approve → status = pending → active (по очереди арбитража)
  - reject  → архив с тегом причины
  - modify  → правка полей, затем approve

если нет ответа за TIMEOUT:
  - эскалировать (другой канал нотификации)
  - продолжать ждать (НЕ auto-reject)

на rejected-цель:
  - мягкий тег для reflect: «эта цель была отклонена, почему?»
  - сигнал в самоподстройку — учится не предлагать заведомо неприемлемое

Решения по operator-approval

  1. Эскалация по таймауту, не auto-reject.
  2. Rejected-цели порождают мягкий сигнал в reflect для самоподстройки агента.
  3. Декомпозиция уже активной цели не требует апрува.

7. ReAct: внутренний цикл #

Базовый вариант

Classic ReAct + Reflexion-style критик и retry-with-memory + опциональный Plan-шаг. Lite Plan-and-Execute: план есть как первоклассный объект траектории, но не обязателен.

Макро-планирование уже сделано в метацикле через декомпозицию в goal_arbitration. ReAct работает по листовой цели. Tactical planning внутри ReAct — отдельное дело, не дублирует декомпозицию.

Структура цикла

trajectory_start(active_goal, focus, memory, allowed_skills)

  [optional] plan_call()             → emit Plan

  loop:
    thought_call(context)            → emit Thought
    if thought signals stop(success/abandon): break

    action_call(context)             → emit Action

    if action.is_irreversible:
      critic_call(action)            → emit Critique
      if critique.verdict == reject: continue   # обратно к thought

    execute(action)                  → emit Observation

    if loop_detector.triggered: break with status=failure
    if budget.exceeded:        break with status=budget_exceeded

  if status == failure and retry_budget > 0:
    add RetryNote → loop again from trajectory_start

  return Trajectory

Внутренние подкомпоненты

Скелет

  • планировщик — генерирует опциональный Plan;
  • внутренний цикл — reasoning ↔ action ↔ observation;
  • критик — § 11;
  • детектор зацикливания — на повторяемости thought/action патернов;
  • бюджет-контроллер — отслеживает потребление, решает «продолжаем / прерываем».

Внешние интерфейсы

  • tool-layer — § 10;
  • retrieval-tool — узкий запрос к memory layer;
  • skill-invoker — переключение режима в рамках allowed_skills;
  • subagent-spawner — § 12;
  • world-model-accessor — запрос явной модели мира через retrieval-tool (KG как один из бэкендов).

Свойства цикла

  • Каждая итерация — две модельные инвокации (thought_call и action_call); критик — отдельный третий вызов на irreversible-действиях. Цена в токенах и латентности компенсируется качеством на слабых локальных моделях.
  • Бюджет жёсткий: превышение → status: budget_exceeded, без молчаливого rollover.
  • Все типы шагов — типизированные записи в Trajectory, не свободный поток текста.
  • Каждый action — через tool-layer; модель не вызывает тулы напрямую.
  • Ошибки тулов — отдельный тип Observation, не выкидывают цикл.
  • Self-критик стреляет только на is_irreversible=true-действиях и в post-failure.
  • Skill-switch внутри цикла разрешён через явный action, только в пределах allowed_skills.
  • Subagent-вызов — отдельный action-type, синхронный.
  • Терминальные статусы: success / failure / budget_exceeded / abandoned.

Решения по ReAct

  1. Базовый вариант — Classic + Reflexion-lite + опциональный Plan.
  2. 2 отдельных модельных вызова за итерацию (thought, action); критик — третий по флагу.
  3. Thought.text — свободный текст, intent-tag не вводим.
  4. Action.expected_outcome — опциональный; чаще заполняется на irreversible-действиях.
  5. Все state-переходы в цикле — типизированные шаги Trajectory.

8. Trajectory и шаги #

Общая обёртка шага

Step {
  id
  type            # thought | action | observation | critique | retry_note | plan
  timestamp
  cost { tokens, latency }
  content         # типизированный payload
  metadata
}

Payload по типам

Thought {
  text
  intent          # опционально: analyze | plan | decide | reflect
}

Plan {
  steps_intended: [{description, expected_tool?}, ...]
  rationale
  revision_of: step_id?    # если пересмотр предыдущего плана
}

Action {
  tool_id
  args                # валидирован по input-schema тула
  is_irreversible     # resolved из tool + skill override
  expected_outcome    # опционально — для калибровки world model
}

Observation {
  outcome             # success | tool_error | schema_error | timeout
                      # | permission_denied | malformed_output | semantic_error
  payload             # валидирован по output-schema тула
  error_detail        # если outcome != success
}

Critique {
  target_step_id      # на какой шаг направлен
  verdict             # ok | warning | reject
  reasoning
  recommendation      # опционально — что делать вместо
  risks_identified: [...]
}

RetryNote {
  previous_trajectory_id
  failure_summary
  recommendation_for_this_attempt
}

Trajectory целиком

Trajectory {
  id
  goal_id
  parent_trajectory_id        # если subagent
  steps: [Step, ...]
  status                      # success | failure | budget_exceeded | abandoned
  final_state                 # полиморфен под predicate_of_success цели:
                              #   structural   → объект под схему
                              #   state_change → ссылка на запрос состояния среды
                              #   judge        → свободный текст
                              #   composite    → агрегирован из детей
                              #   external     → placeholder + ожидаемый сигнал
  total_cost
  started_at, ended_at
}

9. Скиллы #

Скилл — переиспользуемая упаковка компетенции: специализированный промпт + подмножество тулов + параметры. Это процедурная память агента. reflect обновляет именно реестр скиллов.

Схема

Skill {
  id
  name
  description              # для семантического поиска и для LLM
  version
  parent_version_id        # для отката и сравнения версий

  prompt_template          # параметризуемый system prompt
  few_shot_examples: [...]
  allowed_tools: [tool_id, ...]
  required_inputs          # схема входа

  preconditions            # опционально — когда применим
  postconditions           # опционально — что ожидается после

  irreversibility_overrides {
    tool_id: bool | expression    # перекрывает базовое значение тула
  }

  metrics {
    invocation_count
    success_rate
    avg_cost, avg_steps
    failure_modes: {category: count}
    warning_rate
  }

  status                   # active | experimental | deprecated
  origin                   # operator | reflect | imported
  applicable_goal_classes  # [task, inquiry, ...]

  created_at, updated_at
}

Селекция

  • В goal_arbitration: эмбеддинг описания цели → top-K по skill.description, фильтр по applicable_goal_classes, status=active, проверка preconditions.
  • На выход → allowed_skills для goal.

Использование внутри ReAct

  • Один скилл активен в момент. Он диктует system prompt и allowed_tools для всех action'ов.
  • Switch разрешён через явный action switch_skill(skill_id) в пределах allowed_skills. Switch — отдельный шаг в Trajectory.

Лайфцикл

  • Создаётся оператором или reflect.
  • Рефайн → создаётся новая версия, старая → deprecated (не удаляется — нужна для отката и сравнения).
  • Метрики копятся per-version: прямая прослеживаемость улучшений и регрессий.
  • Физическое хранилище — procedural-бэкенд memory-layer'а.

Решения по скиллам

  1. Один активный скилл в ReAct в момент времени; switch — явный action.
  2. Источник is_irreversible: tool (база) → skill (override) → action (resolved).
  3. Скиллы версионируются, deprecated-версии хранятся для сравнения и отката.
  4. Per-version метрики копятся, это вход для reflect при рефайнах.

10. Tool-layer #

Схема тула

Tool {
  id
  name
  description              # для промпта модели

  input_schema             # JSON Schema
  output_schema            # JSON Schema

  base_irreversibility     # never | conditional | always
  conditional_predicate    # если conditional: callable(args) → bool

  side_effects             # human-readable
  category                 # info | action | communication | internal | meta

  retry_policy {
    retryable_outcomes: [transient, timeout, ...]
    max_retries
    backoff: linear | exponential
  }

  timeout
  implementation_ref       # как реально дёргается (http, fn, subagent_dispatcher)

  metrics {
    invocation_count
    success_rate_by_outcome
    avg_latency
    avg_cost
  }
}

Пайплайн обработки action'а

action_emitted_by_react
   ↓
schema_validate(action.args, tool.input_schema)
   fail → Observation(outcome=schema_error, error_detail)
   ↓
resolve_irreversibility(tool, skill, args) → action.is_irreversible
   if true → critic_call (в ReAct, не здесь)
   ↓
invoke(tool.implementation_ref, args) with timeout
   ↓
classify(output)
   → success | tool_error | timeout | permission_denied | malformed_output
   ↓
if retryable_outcome and retries_left:
   retry with backoff
else:
   schema_validate(output, tool.output_schema)
   ↓
Observation(outcome, payload | error_detail) + update tool.metrics

Outcome-таксономия (v1)

OutcomeЗначение
successТул отработал, валидный выход
tool_errorТул упал по внутренней причине
schema_errorНевалидные аргументы или невалидный выход
timeoutТул не уложился в timeout
permission_deniedНет прав на операцию
malformed_outputВыход непарсимый
semantic_errorВыход парсится, но семантически некорректен (детектируется снаружи tool-layer, по постоянным проверкам)

Решения по tool-layer

  1. Реестр тулов — глобальный. Скиллы выбирают подмножество в allowed_tools.
  2. conditional_predicate живёт в spec тула; скилл может перекрыть итог через irreversibility_overrides.
  3. Outcome-таксономия минимальная (6+1 категорий), расширение по факту реальных случаев.
  4. Описания всех тулов из allowed_tools текущего скилла — в system prompt, кешированы на уровне скилла, не меняются динамически по ходу trajectory.
  5. Rate-limits / circuit-breakers — не для v1.

11. Критик #

Когда срабатывает

  • Перед каждым action с is_irreversible=true — основной триггер.
  • После failure всей траектории, если retry_budget > 0 — генерирует RetryNote для следующей попытки.
  • Опционально по явному запросу модели через action request_critique.

Критик НЕ стреляет на каждом шаге — это бесполезно дорого и шумно.

Вход

  • Предлагаемый action (tool, args, expected_outcome);
  • Последние N шагов текущей Trajectory;
  • Активная цель + predicate_of_success;
  • Релевантный фрагмент world model (через retrieval-tool).

Выход

Critique {
  target_step_id
  verdict           # ok | warning | reject
  reasoning
  recommendation    # опционально
  risks_identified: [...]
}

Эффект на цикл

  • ok → action исполняется;
  • warning → action исполняется, verdict логируется, попадает во вход reflect; копится метрика «warning-rate скилла»;
  • reject → action не исполняется, цикл возвращается к thought-call с критикой в контексте.

Решения по критику

  1. Критик — та же модель, другой промпт, явная «роль критика». Если эмпирически увидим коррелированные ошибки — переход на отдельную модель.
  2. Глубина критика — поверхностная: «пострадает ли что-то ценное необратимо, соответствует ли action цели». Глубокая (симуляция, альтернативы) — апгрейд.
  3. Счётчик rejections_per_trajectory + порог → trajectory завершается со status=failure и обычным RetryNote-механизмом.
  4. Warning не блокирует, копится как метрика per-skill; рост — flag для рефактора скилла в reflect.

12. Субагент #

Особый action-type, порождающий дочерний ReAct под подцель. Это не параллельность (v1 синхронен), а иерархическая декомпозиция в рантайме — когда родительский ReAct в процессе понимает, что подзадача достаточно большая для делегирования.

Механика

action: spawn_subagent
args: {
  sub_goal_description
  sub_goal_class
  predicate_of_success
  budget
  allowed_skills          # выбраны вызывающим
}

→ создаётся Goal в goal store; parent_id = current goal; origin = decomposition
→ status = active (без operator-approval, т.к. это не top-level self-цель)
→ Trajectory родителя приостанавливается на этом шаге
→ ReAct запускается на новой Goal (обычной инвокацией)
→ ребёнок выполняется до своего терминала
→ summary ребёнка возвращается родителю как Observation
→ Goal ребёнка обновляется со статусом
→ родитель продолжает с next thought

Формат Observation от ребёнка

{
  child_goal_id
  status                 # success | failure | budget_exceeded | abandoned
  final_state
  total_cost
  summary_text           # короткое; родителю в контекст
}

# Полная child Trajectory доступна по ссылке для post-hoc анализа.
# В контекст родителя не идёт — иначе контекст взорвётся.

Tool-layer аспект

spawn_subagent зарегистрирован как Tool категории meta. Его implementation_ref указывает на subagent_dispatcher — внутренний компонент, который создаёт Goal в store и запускает ReAct.

Решения по субагенту

  1. Sub-goal — реальная Goal-сущность в store (parent_id, origin=decomposition).
  2. Лимит глубины рекурсии = 3.
  3. Лимит активных потомков на родителя = 5.
  4. Параллельные subagent'ы — нет, v1 синхронные.
  5. Observation от ребёнка — сводка, не полный Trajectory.
  6. Бюджет ребёнка задаётся родителем при spawn; бюджет родителя не уменьшается.

13. Memory layer #

Подход

Гибридное хранилище из компонентов; никакой готовый фреймворк не используется, чтобы политика consolidation / retrieval / decay оставалась объектом исследования, а не скрытой деталью реализации.

Бэкенды и индексы

БэкендРеализацияТип памятиНазначение
Episodic store LanceDB (embedded, versioned tables, time-travel) Episodic raw Каждый шаг ReAct — строка; goal_id и timestamp проиндексированы; эмбеддинг в той же таблице. Time-travel даёт бесплатный replay «что агент видел к шагу N».
Vector store Qdrant (local, через docker) Semantic embeddings Семантический retrieval reflection-extracted фактов. Богатый payload filter, чтобы политика консолидации жила в коде, не в БД.
Temporal KG Graphiti на Neo4j (docker) Semantic + World model Bi-temporal knowledge graph. Кастомные Pydantic-сущности → ontology наша. Validity-windows дают «что я знал в момент T».
Procedural store git (bundles) + SQLite (метрики), Voyager-pattern Procedural Скиллы как Python-модули + замороженный промпт + eval-set, версионируются git-тегом. Per-version метрики — таблица в SQLite.
Goal store SQLite — Tree + depends_on, pending_verifications. Отдельно от memory, но логически смежен.

Готовые agent-memory фреймворки (Letta, Mem0, Zep server, Cognee) намеренно отвергнуты: они прячут политики consolidation / decay / retrieval, которые мы исследуем. Reference impls для изучения: MIRIX (memory-manager pattern), Voyager (skill registry pattern).

Единый API

recall(query, types, k, time_range, filters, ...) → MemoryBundle

# types: ['vector', 'graph', 'episodic', 'procedural'] — подмножество
# Возвращает unified bundle с записями из всех запрошенных бэкендов.

# Используется и в meta-recall (грубый запрос с низким k),
# и внутри ReAct через retrieval-tool (узкий запрос с высоким k по теме).

Решения по memory layer

  1. Подход — гибридное хранилище на компонентах, не готовый фреймворк.
  2. Четыре бэкенда: vector / temporal KG / episodic-log / procedural.
  3. Один API recall, параметризованный по типам и фильтрам.
  4. World model хранится в temporal KG (часть memory layer'а).
  5. Тонкая политика consolidation/decay/retrieval — открытая, исследуется отдельно.

14. World model #

Два слоя

  • Неявная (в весах LLM). Всегда присутствует через pre-training. Не редактируема напрямую. Используется по умолчанию там, где явная модель молчит.
  • Явная (поверх неявной). Структурированное представление среды агента. Поддерживается отдельно. В случае конфликта с неявной — в промпте явно ставится приоритет: «факты из контекста перевешивают общее знание».

Что хранится в явной модели

  • Сущности и их состояния (типизированные, с timestamp);
  • Отношения между сущностями;
  • Affordances — какие действия/тулы что меняют;
  • Предсказания и их подтверждения/опровержения (для калибровки);
  • Оценки неопределённости.

Хранение и обновление

  • Темпоральный KG (Graphiti / Zep / своё на Neo4j) — часть memory layer'а.
  • Обновляется в этапе world_update метацикла, после verify:
    • дельта состояния среды после ReAct'а → обновление сущностей и связей;
    • сравнение Action.expected_outcome (если был) с реальностью → запись калибровки.

Использование

Через стандартный recall — KG это один из бэкендов памяти, его выдача попадает в MemoryBundle. Так world model автоматически попадает в каждый LLM-вызов через стандартный контекстный канал, без отдельной коммутации. Внутри ReAct модель может делать узкие запросы через retrieval-tool.

Решения по world model

  1. Явная модель отделена от неявной; в конфликтах приоритет явной через системный промпт.
  2. Хранится в temporal KG, который сам — бэкенд memory layer'а.
  3. Обновляется в отдельном этапе world_update после verify.
  4. В каждый LLM-вызов попадает через recall, без отдельного канала.

15. Reflect #

Что обновляет

  • Реестр скиллов (создание новых версий, deprecation, новые скиллы);
  • Стратегии attention (веса в score-функции, паттерны urgency);
  • Standing-цели (добавление/выталкивание);
  • Семантические факты в KG;
  • Извлечённые абстракции/паттерны.

Триггеры и режимы

ТриггерУсловиеРежим reflect
Volume 5–10 траекторий с прошлого reflect batch_consolidation — паттерны успехов и failure-моды
Event: verify=fail Конкретная неудача failure_analysis — разбор одной траектории
Event: budget exceeded Превышение бюджета budget_postmortem
Event: prediction divergence Большое расхождение expected_outcome и реальности из world_update world_recalibration
Event: novelty relevant Высокий novelty_score в attend, оказавшийся релевантным explanation_extraction в world model
Schedule Раз в ~50 тиков periodic_review — широкий sweep по окну опыта

Почему не каждый тик

Reflect каждый тик имеет несколько серьёзных минусов, не связанных с ценой токенов:

  • N=1 для извлечения паттерна — переобобщение или тривиальный итог;
  • Policy churn — система никогда не стоит на месте, нельзя мерить надёжность отдельной версии (прямо рубит критерий прогресса);
  • Загрязнение семантической памяти низкокачественными извлечениями;
  • Самомодификационное компаундирование — шум reflect'а накапливается в реестре скиллов;
  • Диагностическая непрослеживаемость — изменение в скилле нельзя привязать к конкретному пакету опыта;
  • Дилюция семантики — триггерный reflect имеет фокус («после failures разберись»), reflect каждый тик не знает, на что смотреть.

Решения по reflect

  1. Триггерный, не каждый тик; конкретные триггеры и режимы см. таблицу.
  2. У reflect есть тип/режим, определяемый триггером — даёт фокус промпта и прослеживаемость источника изменений.
  3. Дешёвые скалярные апдейты (счётчики, метрики) не относятся к reflect — они пишутся в store постоянно.
  4. Параметры триггеров (N, K, пороги) конфигурируемы для экспериментов.

16. Сквозные свойства #

Бюджеты

  • Три уровня: на этап, на цель, глобальный (системный).
  • Превышение — событие, не молчаливый rollover.
  • Дефолты задаёт оператор; LLM и reflect могут предлагать; финальное слово — оператор.

Наблюдаемость

  • Каждый этап эмитит trace-event (вход, выход, latency, cost, причина перехода).
  • Каждый шаг ReAct — типизированный объект в Trajectory.
  • Без подробной телеметрии held-out evaluation не даёт причинно-интерпретируемых цифр.

Воспроизводимость

  • Состояние (goal stack + memory snapshot + RNG seed + лог входов sense) — достаточное условие replay'я.
  • Без воспроизводимости честно мерить надёжность нельзя.

Сбои и graceful degradation

  • Любой LLM-вызов может вернуть мусор; retry с backoff на уровне этапа.
  • Если verify не работает — запись «недопроверено», не зацикливание.
  • Если reflect падает — продолжаем без обновления скиллов.

Эволюция

  • Скиллы, политики attention, стратегии арбитража — изменяемые объекты, обновляются через reflect.
  • Версионируются; можно откатить и сравнить версии.

  5. Embeddings — попросить добавить /v1/embeddings на ll-router; fastembed (BGE-M3) как fallback в коде.
  6. Episodic — LanceDB (versioned, time-travel) вместо плоских JSONL-файлов.
  7. Vector — Qdrant в docker.
  8. World model — Graphiti на Neo4j (оба в docker).
  9. Procedural — git (bundles) + SQLite (метрики), паттерн Voyager.
  10. Зависимости — pyproject.toml + uv.lock для воспроизводимости research.
  11. Eval harness — MemoryAgentBench, подключается с v0.
  12. Отвергнуто: Letta, Mem0, Zep server, Cognee (прячут политики); Ollama/vLLM (не нужны); LangGraph/CrewAI/AutoGen (оркестратор пишем сами); Kuzu (архивирован); Memgraph/FalkorDB (не OSI).

18. Журнал решений (сводный) #

Все принятые решения по архитектуре v0.2, в одном месте. Нумерация сквозная.

Метацикл

  1. Цикл синхронный, без прерываний (v1).
  2. Обмен данными между этапами — direct sequential calls + trace-log.
  3. Push-события — в очередь, обрабатываются следующим тиком; рестарта тика нет.
  4. Один ReAct в момент времени; параллелизм — только через spawn_subagent, в v1 синхронный.
  5. Состав этапов: sense → attend → goal_arbitration → recall → react_loop → verify → world_update → [reflect] → store.
  6. Большинство тиков лёгкие; тяжёлая часть (ReAct, reflect) условна.

Подсистема целей

  1. Иерархия — tree + depends_on (DAG-lite).
  2. Бюджеты независимы по целям (вариант A); дефолты — оператор.
  3. Predicate-types — все пять: structural, state_change, judge, composite, external.
  4. Приоритет на structural / state_change для research-валидности.
  5. Aggregation в v1 — all + custom.
  6. Классы — все пять: task, inquiry, maintenance, standing, practice.
  7. Self-цели верхнего уровня — через operator-approval; эскалация по таймауту, не auto-reject.
  8. Verify — immediate-leaf / composite / deferred (через pending_verifications).

Standing-цели

  1. 5–15 standing-целей — «характер» агента.
  2. Reflect-генерируемые standing-цели имеют потолок; новая выталкивает менее эффективную.
  3. Standing-цели только триггерят подцели; сами не действуют.

ReAct

  1. Базовый вариант — Classic + Reflexion-lite + опциональный Plan.
  2. 2 модельных вызова за итерацию (thought + action); критик — третий по флагу.
  3. Thought — свободный текст без обязательного intent-tag.
  4. Action.expected_outcome — опциональный; чаще на irreversible-действиях.
  5. Все state-переходы — типизированные шаги Trajectory.
  6. Goal-decomposition (макро) и tactical planning (внутри ReAct) — два разных уровня.

Скиллы и tool-layer

  1. Один активный скилл в ReAct в момент; switch — явный action.
  2. Источник is_irreversible: tool (база) → skill (override) → action (resolved).
  3. Реестр тулов глобальный; скиллы выбирают подмножество в allowed_tools.
  4. Скиллы версионируются; deprecated-версии остаются для сравнения и отката.
  5. Outcome-таксономия минимальная: 6+1 категорий; расширение по факту.
  6. Описания тулов из allowed_tools — в system prompt, кешируются на скилле.
  7. Rate-limits / circuit-breakers — не для v1.

Критик

  1. Та же модель, другой промпт; явная роль критика. Переход на отдельную модель — апгрейд при коррелированных ошибках.
  2. Поверхностная глубина в v1.
  3. Reject-counter per-trajectory с порогом → failure + RetryNote.
  4. Warning не блокирует, копится как per-skill метрика.

Субагент

  1. Sub-goal — реальная Goal в store (origin=decomposition).
  2. Лимит глубины рекурсии = 3.
  3. Лимит активных потомков на родителя = 5.
  4. Параллельные subagent'ы — нет, v1 синхронные.
  5. Observation от ребёнка — сводка, полный Trajectory доступен по ссылке.
  6. Бюджет ребёнка задаётся родителем при spawn; бюджет родителя на это не уменьшается.

Memory и world model

  1. Гибридное хранилище на компонентах, не готовый фреймворк.
  2. Бэкенды: vector / temporal KG / episodic-log / procedural.
  3. Единый API recall, параметризованный по типам.
  4. World model хранится в temporal KG как часть memory layer'а.
  5. Явная world model имеет приоритет над неявной в промпте.
  6. World model обновляется в этапе world_update после verify.

Reflect

  1. Триггерный, не каждый тик.
  2. Триггеры: volume (5–10) / события (fail, budget, divergence, novelty) / schedule (~50 тиков).
  3. Тип/режим reflect выводится из триггера.
  4. Дешёвые скалярные апдейты — постоянно через store, не через reflect.
  5. Параметры триггеров конфигурируемы.

19. Открытые вопросы #

Не зафиксировано в v0.2

  • Memory layer — детальная политика consolidation, decay, retrieval. Это объект исследования, не реализационная деталь. Отдельный документ.
  • World model — схема узлов KG, кастомные Pydantic-сущности через Graphiti, политика разрешения конфликтов между явным и неявным, формат калибровки предсказаний.
  • Reflect — детальная спецификация каждого режима (что именно меняется в скилле под batch_consolidation, какие сигналы агрегируются в periodic_review, и т.д.).
  • Конкретный набор метрик надёжности и held-out задач поверх MemoryAgentBench. Как численно разделить capability vs reliability.
  • Defaults для бюджетов по классам целей, приоритетам и скиллам. Подбирается эмпирически.
  • Конкретные standing-цели стартового набора. Минимальные 5–10 политик для bootstrapping.
  • Loop-detector — алгоритм. Простой n-gram над action_ids или embedding-similarity между thought'ами; пороги.
  • Recursion depth и siblings лимиты — выбраны эмпирически (3 и 5), но требуется валидация на реальных задачах.
  • Endpoint /v1/embeddings на ll-router — внешнее ожидание; до появления используется fastembed (BGE-M3) локально.

Эти вопросы откладываются в собственные документы по мере спуска в реализацию. Часть из них (consolidation/decay, метрики, режимы reflect) — самостоятельные исследовательские темы.
